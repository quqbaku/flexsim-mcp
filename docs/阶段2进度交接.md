## 3. 当前已知问题（最关键阻塞点）

### 3.1 FlexSimPy `evaluate()` 通道能力不一致（阻塞 Phase 2）

**现象：**

- 节点路径求值可用：
  - 例如 `flexsim_evaluate("MAIN:/project/exec/time")` 返回数值（如 `0.0`）。
- FlexScript 脚本执行疑似不可用：
  - `flexsim_evaluate("return time();")` 返回 `None`（现已被客户端封装为 `""`，但本质仍失败）
  - `createinstance(library().find("Source"), model())` 返回 `None`，导致模板构建失败
  - 任何依赖 `model().subnodes` 的树遍历脚本也返回 `None`

**影响：**

- Phase 2 的核心工具（`flexsim_build_from_template` / `flexsim_add_object` / `flexsim_get_model_tree` 等）都依赖“脚本执行”，因此在当前环境下无法成功构建/查询模型。

**临时缓解（已做）：**

- 对 Phase 1/现有工具链：`FlexSimClient.evaluate()` 对 `None` 不再直接抛错，避免旧工具被异常中断。
- 对 Phase 2 builder：仍要求关键建模脚本返回值（`allow_none=False`），以免出现“误判成功”的假阳性。

**建议的后续方向（下一任必须推进）：**

- 参考 FlexSimPy 官方测试脚本中的 `send/receive` 机制（例如 `TestSendReceive.fsm` 的交互方式），改造 Phase 2 的“脚本执行通道”为：
  - **Python 端**：`controller.send(...)` 发送命令
  - **FlexSim 模型端**：接收消息 → 执行 FlexScript → 回传结果
  - **Python 端**：`controller.receive()` 读取执行结果
- 如果没有一个预置的“执行器模型”（用于接收并执行脚本），则需要先做一次性模型配置并保存一个 `.fsm` 模板，之后 MCP 永远打开它作为执行容器。

### 3.2 “新建空模型”语义仍不彻底

当前 `flexsim_open_model("")` 为避免 GUI 阻塞被设计为“跳过 open() 并提示继续当前模型/提供路径”，因此：

- `flexsim_new_model(path="")` 目前并不能强制得到一个真正“干净空模型”。

后续如果要严格支持“新建空模型”，需要找到一种不会弹窗的可重复流程（取决于 FlexSimPy/引擎侧脚本能力）。

## 4. 最小复现与定位步骤（交接用）

建议接手者先跑以下探针，快速判断当前 FlexSim 环境是否支持 FlexScript：

1. **验证节点路径通道**（应当有数值返回）：
   - 调用：`flexsim_evaluate("MAIN:/project/exec/time")`
2. **验证 FlexScript 通道**（若返回空/None，则 Phase 2 建模无法走 evaluate）：
   - 调用：`flexsim_evaluate("return time();")`
3. **验证模板构建**（当前预期可能失败）：
   - 调用：`flexsim_build_from_template(template_name="simple_production_line")`

若第 2 步失败，建议直接转向 `send/receive` 执行器方案。
