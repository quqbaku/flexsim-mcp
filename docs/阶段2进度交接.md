## 3. 当前已知问题（最关键阻塞点）

### 3.1 FlexSimPy `evaluate()` 通道能力不一致（阻塞 Phase 2）

**现象：**

- 节点路径求值可用：
  - 例如 `flexsim_evaluate("MAIN:/project/exec/time")` 返回数值（如 `0.0`）。
- FlexScript 脚本执行不可用：
  - `flexsim_evaluate("return time();")` 返回 `None`（已封装为 `""`，但本质仍失败）
  - `createinstance(library().find("Source"), model())` 返回 `None`，模板构建失败
  - 任何依赖 `model().subnodes` 的树遍历脚本也返回 `None`

**根本原因（推测）：**

FlexSimPy 的 `evaluate()` 在无 GUI 模式下（`showGUI=False`）可能仅支持节点路径表达式，不支持完整 FlexScript 语句。这是 SDK 设计限制，而非代码 bug。

**临时缓解（已做）：**

- 对 Phase 1 工具：`FlexSimClient.evaluate()` 对 `None` 不再直接抛错，返回空字符串，避免旧工具中断。
- 对 Phase 2 builder：仍要求关键建模脚本返回值（`allow_none=False`），避免误判成功。

### 3.2 `send()/receive()` 通道挂起问题

官方 `TestFlexSimPy.py` 使用 `send/receive` + `TestSendReceive.fsm` 是唯一已知可用的双向通信方案。但：

- 当前 `execute_script()` 的运行时序问题已经修复：现在会先把执行器模型 `reset + run`，再进入 `send/receive`。
- 但最新验证表明：**仅靠一个带 `OnMessage` 触发器、内部调用 `sendtomain(result)` 的 executor.fsm 仍然会超时**。
- 已据此切换到新的验证方向：将回传调用改为 `sendToController(result)`，因为 FlexSimPy 的 `controller.receive()` 等待的是这一条桥接队列，而不是 `sendtomain()`。
- 2026-04-10 当日追加验证：已将 `create_executor_model.py` 和 executor 指南中的触发器代码切换为 `sendToController(result)`，并尝试对现有 `executor.fsm` 原地覆写 `OnMessage` 脚本后重新执行烟测；**结果仍然超时**。
- 同时观察到项目根目录下 `executor.fsm` 的 `LastWriteTime` 没有变化，说明当前环境下通过 `evaluate() + saveModel()` 对现有执行器模型进行自动覆写/保存也没有形成可确认的落盘结果。
- 说明当前问题已经收敛为：**executor.fsm 内部的消息回传协议与 FlexSimPy `controller.receive()` 等待的队列机制不匹配**。

## 5. 最小复现与定位步骤（交接用）

建议接手者先跑以下探针，快速判断当前 FlexSim 环境：

1. **验证官方示例**（最快排查方式）：
   ```
   cd FlexSimPy-25.0.11/TestFlexSimPy
   python TestFlexSimPy.py
   ```
   - 若官方 `TestSendReceive.fsm` 测试通过 → FlexSim 环境正常，问题在 `config.toml` 或路径配置
   - 若挂起/报错 → FlexSim 安装或许可证有问题

2. **验证节点路径通道**（应有数值返回）：
   - 调用：`flexsim_evaluate("MAIN:/project/exec/time")`

3. **验证 FlexScript 通道**（若返回空/None 则 Phase 2 阻塞）：
   - 调用：`flexsim_evaluate("return time();")`

4. **验证模板构建**（预期失败，除非 FlexScript 通道已修复）：
   - 调用：`flexsim_build_from_template(template_name="simple_production_line")`

若第 3 步失败，**立即转向手动创建 `executor.fsm` 方案**（参考 `docs/executor-fsm-guide.md`）。

补充：若 `executor.fsm` 已存在但 `execute_script()` 仍超时，则说明当前 executor 模型的 `OnMessage + sendtomain(result)` 方案并不足以打通回传链路。当前优先验证的新方案是 `OnMessage + sendToController(result)`。

补充验证结论：当前仓库里已尝试将脚本侧切到 `sendToController(result)`，但由于自动覆写 `executor.fsm` 未形成可确认的文件更新时间变化，且 `smoke_eval.py` 仍失败，所以**方案 1 尚未解除阻塞**。下一步应转向在 FlexSim GUI 中直接检查/修改官方或手工 executor 模型，而不是继续依赖 `evaluate()` 自动写回模型。
